**User** : สร้างโปรแกรม ai chat bot เพื่อตอบลูกค้าที่เกี่ยวกับข้อมูลการขายโครงการคอนโดมิเนียม มี requirements ดังนี้
*  รองรับการ chat ได้พร้อมกันหลายคน
*  สามารถจดจำลูกค้าแต่ละคนได้ โดยไม่ต้อง log in
*  เก็บประวัติการคุยของแต่ละคนได้ และจดจำได้ว่าคุยอะไรไปแล้ว
*  พัฒนาในรูปแบบ web app
*  ใช้ข้อมูลในการ chat จากโฟลเดอร์ชื่อ info
*  แอดมินจะเป็นคนเตรียมข้อมูลใน info ทั้งหมด จะมีทั้งไฟล์แบบ txt, docx, xlsx, jpg, mp4, pdf
*  ถ้ามีการคุยกันนอกหัวข้อใน info ให้ตอบว่า "ไม่สามารถตอบได้"
*  เป้าหมายสุดท้ายในการ chat คือ นัดหมายเยี่ยมชม sale gallery โดยเก็บ ชื่อ นามสกุล อีเมล เบอร์โทรศัพท์
*  ส่งข้อมูลการนัดหมายไปที่ telegram bot อัตโนมัติ
*  โปรแกรมทั้งหมดอยู่ใน docker
**Gemini** : การสร้าง AI Chatbot สำหรับโครงการคอนโดมิเนียมตาม Requirements ที่คุณกำหนดมานั้น ถือเป็นระบบที่มีความสมบูรณ์และพร้อมใช้งานจริง (Production-ready) ครับ เพื่อให้ตอบโจทย์ทั้งหมด เราจะใช้ Stack เทคโนโลยีดังนี้:
*   **Backend & Bot Framework:**   **Python (FastAPI)**  เหมาะสำหรับการรองรับ Concurrency (คุยพร้อมกันหลายคน) และจัดการ Session
*   **AI & RAG:**   **LangChain / LlamaIndex + OpenAI API**  (หรือ LLM อื่นๆ) เพื่ออ่านไฟล์ในโฟลเดอร์ info และจำกัดขอบเขตการตอบ (Strict Guardrails)
*   **Data Parsing:**  Unstructured หรือไลบรารีเฉพาะทาง (เช่น PyPDF2, python-docx, openpyxl, OCR/Whisper สำหรับรูปภาพและวิดีโอ)
*   **Session & History:**   **Redis**  เพื่อจดจำผู้ใช้แยกตาม Session ID (จาก Browser Cookies/Local Storage) โดยไม่ต้อง Login
*   **Deployment:**   **Docker & Docker Compose**

เนื่องจาก Requirement ระบุว่ามีไฟล์หลายประเภท แอดมินอาจจะอัปโหลดเข้าไปทั้งหมด ระบบ RAG ทั่วไปจะอ่านได้ดีแค่ .txt, .pdf, .docx หากต้องการให้ AI เข้าใจไฟล์ประเภทอื่นด้วย มีแนวทางปฏิบัติดังนี้ครับ:
1.  **ไฟล์**  **.xlsx**  **(ตารางราคา/ห้องว่าง):**  แนะนำให้เขียนสคริปต์ Python แปลงจาก Excel ให้เป็นตารางรูปแบบ Markdown Text ก่อนส่งให้ AI อ่าน จะทำให้ AI เข้าใจโครงสร้างตารางและตอบราคาได้แม่นยำที่สุด
2.  **ไฟล์**  **.jpg**  **(โบรชัวร์/Floor Plan):**  ควรเปิดใช้งาน  **OpenAI Multimodal (เช่น gpt-4o)**  เพื่อให้ AI สามารถ "มองเห็น" รูปภาพได้ หรือใช้ Python ทำ OCR ดึงข้อความในภาพออกมาก่อนพิมพ์เก็บไว้
3.  **ไฟล์**  **.mp4**  **(วิดีโอรีวิวห้อง):**  ควรใช้  **OpenAI Whisper API**  ในการแปลงเสียงพูดในวิดีโอออกมาเป็น "ซับไตเติ้ล Text" แล้วเซฟเป็น .txt เก็บไว้ในโฟลเดอร์ เพื่อให้ AI นำเนื้อหาในวิดีโอไปตอบลูกค้าได้ครับ

##### 3. สถานการณ์จำลองสำหรับทดสอบ (Test Cases)
เพื่อให้แน่ใจว่า Requirements ทั้งหมดทำงานได้สมบูรณ์แบบ แนะนำให้คุณลองกด Execute ทดสอบพิมพ์ข้อความตาม 4 สถานการณ์นี้ครับ:
*   **สถานการณ์ที่ 1: ดึงข้อมูลจากไฟล์ (RAG)**  พิมพ์ถามข้อมูลที่มีอยู่ในไฟล์โฟลเดอร์ info เช่น  *"มีห้องแบบกี่ตารางเมตรบ้าง?"*  หรือ  *"ราคาเริ่มต้นเท่าไหร่?"*  เพื่อเช็คว่า AI ดึงข้อมูลมาตอบได้ถูกต้องหรือไม่
*   **สถานการณ์ที่ 2: ป้องกันการออกนอกเรื่อง (Guardrails)**  พิมพ์ถามเรื่องทั่วไปที่ไม่เกี่ยวกับคอนโด เช่น  *"วันนี้กินข้าวกับอะไรดี?"*  หรือ  *"สภาพอากาศวันนี้เป็นอย่างไร?"*  บอทจะต้องตอบกลับมาแค่คำว่า  **"ไม่สามารถตอบได้"**  เท่านั้น
*   **สถานการณ์ที่ 3: ทดสอบความจำ (Session Memory)**  พิมพ์แนะนำตัวไปก่อน เช่น  *"สวัสดี ฉันชื่อเอกนะ"*  กด Execute 1 รอบ แล้วเปลี่ยนคำถามใหม่เป็น  *"เมื่อกี้ฉันบอกว่าฉันชื่ออะไรนะ?"*  เพื่อเช็คว่า Redis จดจำผู้ใช้ได้ถูกต้อง
*   **สถานการณ์ที่ 4: การเก็บข้อมูลนัดหมาย (Lead Capture)**  พิมพ์บอกบอทว่า  *"สนใจเข้าไปดูห้องจริงครับ"*  บอทควรจะเริ่มทำตาม Prompt โดยการทยอยขอข้อมูลคุณ (ชื่อ-นามสกุล, อีเมล, เบอร์โทร) ทีละอย่าง เมื่อคุณให้ข้อมูลครบ ระบบควรจะสรุปข้อมูลให้ครับ

##### 🧪 วิธีที่ 1: ใช้โหมดไม่แสดงตัวตน (Incognito Mode) 🌟 แนะนำง่ายที่สุด

3.  **ทดสอบความจำแยกคน:**  * กลับไปที่หน้าต่างของ  **สมชาย**  แล้วพิมพ์ถามว่า  *"เมื่อกี้ผมบอกว่าผมชื่ออะไรนะ?"*  บอทต้องตอบว่า  **สมชาย**

*  content_preview: ข้อความที่มันอ่านและแกะออกมาได้จริงๆ หน้าตาเป็นอย่างไร (ช่วยให้รู้ว่า OCR แกะภาษาไทยเพี้ยนไหม หรือ Web Scraper กวาดตารางมาติดขยะมาด้วยหรือเปล่า)
**User** : จากผล debug พบว่า เข้าไปดึงข้อมูลจากเว็บไซต์ไม่ครบ ดึงมาแค่ <title> ต้องแก้ยังไง
**Gemini** : อาการดึงข้อมูลมาได้แค่ <title> เป็นปัญหาที่พบบ่อยมากในการทำ Web Scraping ร่วมกับ RAG ครับ โดยเกิดจาก  **2 สาเหตุหลัก**  ดังนี้ครับ:
1.  **โดนเซิร์ฟเวอร์บล็อก (Bot Protection / User-Agent Block):**  เว็บไซต์ส่วนใหญ่ (โดยเฉพาะเว็บโครงการคอนโด หรือเว็บอสังหาริมทรัพย์) จะมีระบบป้องกันบอท หากเรายิงเข้าไปตรงๆ โดยไม่ระบุตัวตน เซิร์ฟเวอร์จะรู้ว่าเป็นบอท Python/LangChain และจะส่งกลับมาแค่หน้าเปล่าๆ หรือหน้า Error ซึ่งมักจะมีแค่โครง HTML และ <title> เท่านั้น
2.  **เว็บไซต์เป็น JavaScript Rendering (SPA):**  หน้าเว็บสมัยใหม่ที่สร้างด้วย React, Vue หรือ Next.js ตัวเนื้อหาจริงจะถูกโหลดผ่าน API หลังจากเปิดหน้าเว็บไปแล้ว Loader แบบธรรมดาจึงมองเห็นแค่โครงหน้าเว็บว่างๆ
เรามาเริ่มแก้ไขจากสาเหตุแรก ซึ่งเป็นจุดที่เจอบ่อยที่สุด โดยการ  **"ใส่ User-Agent เพื่อปลอมตัวเป็นเบราว์เซอร์จริง"**  ครับ

##### 🌐 ทางเลือกที่ปลอดภัยกว่าการ Forward Port
**1. ใช้ Cloudflare Tunnel (แนะนำที่สุดสำหรับรันเครื่องตัวเอง หรือ Home Server)**  หากคุณรันระบบนี้บนเครื่องตัวเอง หรือแม้แต่รันผ่าน Docker บน Synology NAS วิธีที่ปลอดภัยและฟรีคือการติดตั้ง  **Cloudflare Tunnel (cloudflared)**  ครับ
*   **ข้อดี:**  ไม่ต้องยุ่งกับการตั้งค่าเร้าเตอร์ ไม่ต้องเปิด Port ระบบจะสร้างท่อที่เข้ารหัสข้อมูลจากเครื่องคุณไปที่เซิร์ฟเวอร์ Cloudflare ให้โดยอัตโนมัติ
*   **ความปลอดภัย:**  ได้โดเมน HTTPS (กุญแจเขียว) ฟรีทันที และยังได้ระบบป้องกัน DDoS ของ Cloudflare มาช่วยบังหน้าให้ด้วย
**2. ใช้ Ngrok (เหมาะสำหรับเทสชั่วคราวแบบด่วนๆ)**  ถ้าแค่อยากส่งลิงก์ให้ทีมงานหรือลูกค้าลองเล่นเดี๋ยวนี้เลย การรันคำสั่ง ngrok http 8000 ใน Terminal จะเป็นวิธีที่ไวที่สุดครับ
*   **ข้อดี:**  รันคำสั่งเดียวจบ ได้ลิงก์ชั่วคราว (เช่น https://abc-123.ngrok.io) ส่งให้คนอื่นเล่นได้เลย
*   **ข้อจำกัด:**  ลิงก์จะเปลี่ยนไปเรื่อยๆ ทุกครั้งที่เปิดใหม่ (สำหรับเวอร์ชันฟรี) และความเร็วอาจจะสู้การตั้งค่าเซิร์ฟเวอร์จริงๆ ไม่ได้
**3. นำขึ้น Cloud Server (เหมาะสำหรับ Production ใช้งานจริง)**  หากแชทบอทตัวนี้พร้อมให้บริการลูกค้าจริง การนำไฟล์โปรเจกต์และ Docker Compose ที่เราทำไว้ไป Deploy บน Vultr หรือเซิร์ฟเวอร์ Cloud อื่นๆ จะตอบโจทย์ที่สุดครับ
*   **ข้อดี:**  ระบบทำงานได้ตลอด 24 ชั่วโมงโดยไม่ต้องเปิดเครื่องคอมพิวเตอร์ทิ้งไว้ และสามารถใช้ Portainer บริหารจัดการ Container ได้อย่างเป็นระบบ
*   **ความมั่นคง:**  ได้ Public IP ของตัวเองอย่างชัดเจน และสามารถผูกโดเมนเนมของโครงการเข้าไปได้อย่างสมบูรณ์

*  กดปุ่ม  **"อัปเดตไฟล์"**  ที่หน้า Dashboard แอดมิน (หรือสั่งรีสตาร์ท Docker) เพื่อให้ระบบสูบข้อมูลใหม่เข้าไปใน FAISS Vector Store
##### 2. ปรับแต่ง "วิญญาณ" ของบอท (System Prompt) 🌟 จุดนี้สำคัญมาก
ในไฟล์ app/chatbot.py เราได้เขียนคำสั่งฝังเอาไว้ (Hardcode) ว่าน้องแอนคือ  **"เซลล์ขายคอนโดมิเนียม"**  และมีเป้าหมายคือการชวนลูกค้ามาดู  **"ห้องตัวอย่าง"**  หากโครงการใหม่ของคุณเป็นโปรเจกต์แนวราบหรือบ้านระดับลักชัวรีอย่าง Monsane Exclusive Villa คุณจะต้องเปิดไฟล์ app/chatbot.py แล้วไปแก้ข้อความในตัวแปร SYSTEM_PROMPT ด้วยครับ ไม่อย่างนั้นน้องแอนอาจจะเผลอขายวิลล่าหรูแต่ดันเรียกตัวเองว่าเซลล์คอนโดได้ครับ
**ตัวอย่างจุดที่ต้องแก้ใน**  **SYSTEM_PROMPT**  **:**
*  เปลี่ยนคำว่า  *"เซลล์ขายคอนโดมิเนียม"*  ->  *"ที่ปรึกษาการขายโครงการบ้านเดี่ยวลักชัวรี"*
*  ปรับโทนเสียงให้เข้ากับกลุ่มเป้าหมาย (เช่น ถ้าโครงการราคาเริ่มต้นสูงมาก อาจจะต้องลดความ "ออดอ้อน" ลง และเพิ่มความ "เป็นทางการ/มืออาชีพ" ขึ้น)
*  เปลี่ยนเป้าหมายจาก  *"เยี่ยมชมห้องตัวอย่าง"*  ->  *"นัดหมายเยี่ยมชมโครงการส่วนตัว"*
##### 3. ตรวจสอบเงื่อนไขการเก็บ Lead
หากโครงการใหม่นี้ คุณยังคงต้องการให้บอทเก็บข้อมูล  **ชื่อ-นามสกุล, อีเมล, และเบอร์โทรศัพท์**  3 อย่างเหมือนเดิม คุณก็ไม่ต้องแก้โค้ดส่วน Regex หรือการส่งเข้า Telegram เลยครับ ระบบคัดกรองเบอร์โทรศัพท์และอีเมลจะยังคงทำงานได้ดีเยี่ยมเหมือนเดิม
เมื่อปรับเปลี่ยนไฟล์และ System Prompt เสร็จแล้ว อย่าลืมเข้าไปที่ http://localhost:8000/api/admin/debug-rag เพื่อตรวจสอบดูว่าข้อมูลโครงการใหม่ถูกตัดแบ่งและโหลดเข้าไปในระบบอย่างครบถ้วนหรือไม่ครับ
**User** : ต้องการให้เก็บข้อมูลจากหน้าเว็บทุกหน้า โดยการ crawling ไปตามลิงค์ต่างๆ และ ไปตาม sitemap ต้องทำอย่างไร
**Gemini** : การให้บอทวิ่งเก็บข้อมูลทั้งเว็บไซต์ (Web Crawling & Spidering) เป็นไอเดียที่ทรงพลังมากครับ! แต่ในทางปฏิบัติสำหรับระบบ RAG (แชทบอท AI) ผมขอแนะนำให้ใช้  **การดึงจาก Sitemap (Sitemap Crawling)**  เป็นหลัก และหลีกเลี่ยงการให้บอทวิ่งมั่วซั่วตามลิงก์ (Recursive Crawling) ด้วยเหตุผลสำคัญ 2 ข้อครับ:
1.  **ปัญหาขยะล้นสมอง (Garbage Data):**  ถ้าเราให้บอทวิ่งตามทุกลิงก์ มันจะไปเก็บหน้า "นโยบายความเป็นส่วนตัว", "เงื่อนไขการใช้งาน", หรือหน้าข่าวเก่าๆ เมื่อ 5 ปีที่แล้วมาด้วย ซึ่งจะทำให้ AI สับสนและตอบคำถามลูกค้าเพี้ยนได้
2.  **เปลืองทรัพยากร:**  การให้ Playwright เปิดเบราว์เซอร์ล่องหนทุกหน้าของเว็บพร้อมๆ กัน อาจจะทำให้ RAM ของเครื่องเซิร์ฟเวอร์เต็มและระบบล่มได้ครับ
ดังนั้น วิธีที่  **ปลอดภัย แม่นยำ และเป็นมืออาชีพที่สุด**  คือการให้บอทอ่านไฟล์ sitemap.xml ของเว็บไซต์ แล้วดึงเฉพาะหน้าเว็บที่มีอยู่จริงมาเข้าคิวให้ Playwright ไปสูบข้อมูลครับ

##### 🛠️ จุดที่ 1: ไขปริศนาบั๊ก slice indices must be integers
ในโค้ดเดิม คุณเขียนเงื่อนไขว่า elif file.endswith(".xlsx", ".xls"): และ elif file.endswith(".docx", ".doc"):
*   **สาเหตุที่พัง:**  ในภาษา Python คำสั่ง .endswith() จะรับค่าตัวแปรตัวที่ 2 เป็น "ตัวเลขตำแหน่งเริ่มต้น (Start Index)" ครับ พอเราใส่คำว่า ".xls" (ซึ่งเป็นตัวอักษร) ลงไปเป็นตัวที่ 2 Python เลยงงและพ่น Error ว่า  *"slice indices ต้องเป็นตัวเลขสิ!"*  ออกมาครับ
*   **ทำไมอ่าน .mp4 ถึงพังด้วย?:**  เพราะโค้ดมันทำงานจากบนลงล่าง พอหยิบไฟล์ .mp4 มา มันไปเช็คเงื่อนไข .docx ก่อนและเจอโครงสร้างที่ผิด มันเลย Error ทันทีโดยที่ยังไม่ทันตกถึงเงื่อนไข .mp4 เลยครับ!
*   **วิธีแก้:**  ต้องใส่วงเล็บซ้อนเข้าไปให้เป็น Tuple ครับ เช่น file.endswith((".xlsx", ".xls"))
##### 🛠️ จุดที่ 2: การย่อหน้า (Indentation) ผิดพลาด
ในฟังก์ชัน fetch_urls_safely บรรทัดที่เขียนว่า page.wait_for_timeout(3000) มีการใช้ปุ่ม Tab หรือเว้นวรรคไม่ตรงกับบรรทัดอื่น ซึ่งจะทำให้เปิดเซิร์ฟเวอร์ไม่ขึ้น (IndentationError) ผมได้จัดช่องว่างให้ตรงกันเรียบร้อยครับ
##### 🛠️ จุดที่ 3: ตัวเลขในฟังก์ชัน Debug ไม่ตรงกัน
ในฟังก์ชัน get_vectorstore_debug_data เราตัดข้อความมาแสดงที่ 1500 ตัวอักษร [:1500] แต่เงื่อนไขแสดงจุดไข่ปลา ... ยังเขียนเป็น > 300 อยู่ ผมปรับให้เป็น > 1500 ให้สอดคล้องกันแล้วครับ

##### 💻 โค้ด chatbot.py ฉบับแก้ไขสมบูรณ์ (คัดลอกไปทับได้เลย)
นำไปเซฟทับไฟล์เดิมแล้วสั่ง docker-compose restart web-chatbot ระบบจะสามารถอ่านไฟล์ Excel สกุลต่างๆ และไฟล์ .mp4 ได้อย่างราบรื่นโดยไม่เกิด Error รบกวนอีกแน่นอนครับ! ลองทดสอบดูได้เลยครับ
**User** : ดูจาก debug แล้ว  ข้อมุลจากไฟล์ .xlsx ถูกอ่านแล้ว แต่ chatbot ไม่เอาข้อมูลที่ได้มาตอบ
**Gemini** : อาการแบบนี้เป็น "Pain Point" สุดคลาสสิกของการทำระบบ RAG (แชทบอทอ่านไฟล์) เลยครับ!
สาเหตุที่ในหน้า Debug (FAISS Vector Store) มีข้อมูลอยู่ครบ แต่พอลูกค้าถามแล้วบอทตอบไม่ได้ เกิดจาก  **2 สาเหตุหลัก**  ครับ:
1.  **โควต้าการดึงข้อมูลน้อยเกินไป (k-value):**  ในโค้ดปัจจุบันเราตั้งค่าให้บอทดึงข้อมูลที่ "คิดว่าเกี่ยวข้องที่สุด" มาแค่  **3 ชิ้น (k=3)**  ถ้าระบบมีข้อมูลจากหน้าเว็บและไฟล์อื่นๆ เยอะ ข้อมูลจาก Excel อาจจะถูกจัดอันดับความเกี่ยวข้องตกไปอยู่อันดับที่ 4 หรือ 5 ทำให้บอทมองไม่เห็นข้อมูลนั้นตอนตอบคำถามครับ
2.  **AI อ่านตารางไม่เก่ง (Markdown Table):**  ข้อมูล Excel ถูกแปลงเป็นตาราง (มีขีด |---|) ซึ่งบางครั้ง AI จะกวาดสายตาข้ามไปเพราะคิดว่าไม่ใช่ข้อความ หรือจับใจความ (Semantic) ไม่ตรงกับคำถามของลูกค้า
เราสามารถแก้ปัญหานี้ได้ง่ายๆ ด้วยการ  **"เพิ่มโควต้าการมองเห็น"**  และ  **"สั่งกำชับใน Prompt ให้อ่านตาราง"**  ครับ
